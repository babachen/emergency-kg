# 系统设计说明

## 一、系统定位
本系统面向“基于大语言模型的应急预案知识图谱构建”课题答辩展示，采用“业务管理 + 图谱服务 + AI 适配层”的三层思路：
- MySQL：基础业务库与图谱明细存储
- Neo4j：图数据库查询增强层，支持版本快照同步
- bysj-chat-api：可选真实大模型接口

## 二、核心业务对象
- 省市区域 region
- 预案文档 plan_document
- 文档分段 plan_section
- 抽取任务 extraction_task
- 实体类型 entity_type
- 关系类型 relation_type
- 图谱实体 kg_entity
- 图谱关系 kg_relation
- 三元组 kg_triple
- 冲突记录 knowledge_conflict
- 补全建议 knowledge_completion
- 图谱版本 graph_version
- AI 调用记录 ai_call_record

## 三、权限角色设计
### 1. 系统管理员
- 用户/角色/菜单管理
- 全部业务数据可见可编辑
- 版本发布与系统配置

### 2. 知识分析员
- 查看预案、任务、图谱、冲突、补全
- 使用问答推理与图谱检索功能
- 参与知识分析与人工校验

### 3. 数据维护员
- 上传预案、维护文档内容
- 执行预处理和抽取任务
- 查看图谱实体与质量结果

## 四、关键闭环
### 1. 预案入库闭环
上传附件 / 补录正文 -> 保存预案 -> 文本分段 -> 状态更新

### 2. 抽取闭环
创建抽取任务 -> 执行任务 -> 调用 AI 适配层 -> 生成实体/关系/三元组 -> 冲突检测 -> 补全建议

### 3. 分析闭环
图谱检索 -> 可视化 -> 知识问答 -> 规则推理 -> 版本快照 -> Neo4j 同步

## 五、后续可扩展点
1. 接入 Word/PDF 正文解析
2. 基于真实省级预案来源的批量采集与自动清洗
3. 多轮问答与来源证据高亮
4. 质量评估指标细化（准确率、覆盖率、冲突率、补全增益）
5. 面向 31 省的批量导入与数据采集脚本
